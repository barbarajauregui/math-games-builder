## §1 — Summary

This plan ships four small additive changes to four existing game engines so that two foundational Kindergarten standards — K.CC.A.1 ("count to 100 by ones and tens") and K.CC.A.2 ("count forward beginning from a given number") — have clean, unconditional PRIMARY engine coverage before the cross-age pilot launches. Each fix adds a new mode or config flag to an engine that already exists; no engine is being built from scratch and no existing scenarios should break. Realistic total effort is **5–8 implementation hours plus 2 hours of testing and paperwork**. Recommended sequencing: split into two sessions — Fix 2 + Fix 3 together (both touch the same Phaser scene, Fix 3 is trivial once Fix 2 lands), then Fix 4 + Fix 5 together in a second session (different engines, fully independent of the first pair).

## §2 — Per-fix sections

### Fix 2 — `hundred-board` count-trail mode

**Engine in plain English:** The hundred-board engine puts a 10×10 grid of numbered cells on screen (1 through 100). Today it has one mode: "find-cell" — show the kid a target number and they tap the matching cell. We're adding a second mode: "count-trail" — the kid taps cells in sequence starting from 1, and a glowing trail lights up behind them as they go. If they tap a cell out of sequence, the trail resets. Win when they reach the target with no skips.

#### Scope

- **Add** a new mode to `HundredBoardScene`. Mode is chosen by a config flag: `mode: 'find-cell' | 'count-trail'`. Default `'find-cell'` so existing scenarios keep working unchanged.
- **In count-trail mode:** scene tracks `expectedNext` (starts at 1). Each tap is checked against `expectedNext`. Correct → cell joins the trail (gets the glow style), `expectedNext` increments. Incorrect → trail clears, `expectedNext` resets to 1, kid sees a gentle "start again from 1" cue. Round ends when `expectedNext - 1 === target`.
- **New round-data shape:** count-trail rounds need `{ mode: 'count-trail', target: number }`. Find-cell rounds keep their existing shape.
- **New game option IDs** in the registry so a Builder can select count-trail as a distinct game option (not just a config knob hidden from them).
- **Stays the same:** the find-cell mode, all existing scenarios that use it, the visual style of cells, the win-screen.

#### Files to touch

- `src/lib/game-engines/montessori-phaser.ts` — add count-trail branch inside `HundredBoardScene`. New private state: `expectedNext`, `trailCells: Set<number>`. New tap handler logic. New trail-render method.
- `src/lib/game-engines/game-option-registry.ts` — add a new game option ID `hundred-board-count-trail` (or similar) with display name, description, and mapping to scene config.
- Engine-router glue inside `montessori-phaser.ts` (or wherever scene + option-ID resolve into a config object) — make sure the new option ID routes to `HundredBoardScene` with `mode: 'count-trail'`.

#### Implementation steps

1. Read the existing `HundredBoardScene` start-to-finish to map current state, tap handler, and round-completion path.
2. Add config field `mode` to whatever shape the scene receives at `init`/`create` time. Default to `'find-cell'`.
3. Add new private state: `expectedNext: number = 1`, `trailCells: Set<number> = new Set()`.
4. Add a `renderTrail()` private method — for each cell number in `trailCells`, apply a glow tint (use the existing tint mechanism for highlight cells if there is one; otherwise add a Phaser tween/filter).
5. Branch the tap handler on `this.mode`:
   - `'find-cell'`: existing behavior, unchanged.
   - `'count-trail'`: if tapped cell number === `expectedNext` → add to `trailCells`, increment `expectedNext`, re-render trail, check win (`expectedNext - 1 === target` → fire win). Else → clear `trailCells`, reset `expectedNext` to 1, re-render trail, play a "start again" cue.
6. Add the new game option ID in the registry with description like "Count from 1 up to the target by tapping cells in order. The trail glows behind you."
7. Wire option-ID → scene config in the engine router so picking the new option spins up `HundredBoardScene` with `mode: 'count-trail'` and a target.

#### Risks

- **Silent misrouting:** if option-ID → scene-config plumbing isn't updated, picking the new option could launch find-cell mode and look broken with no error. Mitigation: explicit test of "select new option → kid sees count-trail behavior" in the test plan.
- **Existing scenarios breaking:** if the default `mode` isn't `'find-cell'` everywhere, scenarios that don't pass `mode` could start in count-trail mode. Mitigation: explicit default in scene init; manually run one existing find-cell scenario after the change.
- **Reset feels punishing:** if a kid skips one and the trail clears with no warning, it feels bad. Mitigation: visible reset cue (brief shake or color flash) plus a "start again from 1" hint.
- **Performance:** trail of up to 100 glowing cells should be fine on a Chromebook, but a tween-per-cell could stutter. Mitigation: prefer a single redraw of cell tints over per-cell tweens.

#### Test plan

- TypeScript build: `npm run build` clean.
- Manual: launch a count-trail round with target = 10. Tap cells 1 through 10 in order — trail lights up, win fires.
- Manual: launch count-trail with target = 10. Tap 1, 2, 4 — trail clears at the skip; tap 1 through 10 cleanly; win fires.
- Manual: launch an existing find-cell scenario — behavior identical to before.
- Manual: launch count-trail with target = 30 — trail renders correctly across multiple rows.

#### Estimated effort

**3–4 hours** (real new scene state, trail rendering, registry + router work, manual testing).

#### Unlocks

- K.CC.A.1 PRIMARY engine `hundred-board` flips from "conditional — needs count-trail mode" to clean.
- Rationale string for K.CC.A.1 in `src/data/standard-mechanic-map.json` drops the conditional caveat.

### Fix 3 — `hundred-board` start-from-N variant

**Engine in plain English:** Same count-trail mode from Fix 2, but instead of starting at 1, the kid starts at any cell N. They tap 47, 48, 49, 50 ... up to the target. This is what K.CC.A.2 requires ("count forward beginning from a given number other than 1").

#### Scope

- **Add** a `startCell: number` config option to count-trail mode. Default `1` so Fix 2 scenarios are unaffected.
- **In count-trail mode with startCell = N:** `expectedNext` initializes to `N`. First tap must be cell `N`. Otherwise behavior identical to Fix 2.
- **Stays the same:** trail rendering, reset behavior, win condition (kid reaches target).
- **New game option ID** (or new config on the count-trail option) so this is a Builder-selectable variant.

#### Files to touch

- `src/lib/game-engines/montessori-phaser.ts` — initialize `expectedNext` from `config.startCell ?? 1` inside count-trail branch.
- `src/lib/game-engines/game-option-registry.ts` — add `hundred-board-count-on-from-n` (or extend the count-trail option with a `startCell` parameter the Builder picks).

#### Implementation steps

1. Add `startCell: number` to count-trail config shape, default 1.
2. In the count-trail `init` path, set `this.expectedNext = config.startCell`.
3. Add the new game option ID (or extend Fix 2's option) so Builders can pick "count on from 47" vs "count from 1."
4. Wire the new option to scene config in the engine router.

#### Risks

- **Win condition off-by-one:** if Fix 2 used `expectedNext - 1 === target` and target is now the END number (not the count), make sure semantics still hold when starting from N. (They do — kid needs to tap target last, so win still fires when `expectedNext - 1 === target`.)
- **Builder UX confusion:** "start from" + "target" needs to be clear in the option config UI. Mitigation: copy in the registry entry should spell it out — e.g., "Kid starts at cell 47 and counts up to cell 60."

#### Test plan

- TypeScript build clean.
- Manual: count-trail with `startCell = 47`, target = 60. Tap 47 through 60 in order, win fires.
- Manual: same config, tap 47, 48, 50 — trail resets to start-fresh at 47.
- Manual: count-trail with no `startCell` set — behaves identically to Fix 2 (starts at 1).

#### Estimated effort

**~15 minutes** once Fix 2 is solid (this is genuinely just a config flag on top of Fix 2's machinery).

#### Unlocks

- K.CC.A.2 PRIMARY engine `hundred-board` flips from "conditional — needs start-from-N variant" to clean (one of the three engines that need fixing for this standard).

### Fix 4 — `number-frames` counting-on mode

**Engine in plain English:** Number frames is the ten-frame engine — a 2×5 grid where kids tap to fill in dots. Today every round starts empty and kids count from 1. We're adding "counting on" — the frame starts with N dots already drawn in, and the kid's first tap counts as N+1, not 1. This is the ten-frame way of teaching "count forward from a given number."

#### Scope

- **Add** a `prefillCount: number` config to the existing AddRound (or equivalent round-type that draws dots into a frame). Default 0.
- **At round start:** the engine renders `prefillCount` dots in `frame-a` (the first ten-frame) before any kid interaction. The internal counter is initialized to `prefillCount`. Each tap increments by 1.
- **Stays the same:** all existing rounds with `prefillCount = 0` (i.e., everything that exists today) behave unchanged. Visual style of dots is identical for prefilled and tapped-in dots.
- **New round set** OR **new GameOption** for `number-frames-count-on` — pick whichever fits the existing pattern in the registry (Way 1 agent flagged both as valid; the implementer should pick the lower-friction option based on what the registry already supports).

#### Files to touch

- `src/lib/game-engines/number-frames.ts` — extend AddRound (or the relevant round variant) to accept `prefillCount`. Pre-fill `frame-a` with that many dots at round start. Initialize the internal count to `prefillCount`.
- `src/lib/game-engines/game-option-registry.ts` — add `number-frames-count-on` option (or new round set) with display name, description.
- If a separate round-set file exists (e.g. `ROUNDS_COUNT_ON`), add it next to the existing round sets.

#### Implementation steps

1. Read `number-frames.ts` and identify the AddRound shape and where rounds are configured.
2. Add `prefillCount?: number` to the round/config type. Default 0 if not provided.
3. At round start, render `prefillCount` dots into `frame-a` using the same render path tap-dots use. Initialize the counter to `prefillCount`.
4. Verify the win-check still uses the total count (prefill + taps) against the target — should already be correct if you initialized the counter properly.
5. Register the new option/round-set in `game-option-registry.ts`.

#### Risks

- **Prefilled dots look different from tapped dots:** if they're drawn differently, the kid might think they don't count. Mitigation: use the exact same render path.
- **Counter init bug:** if you set the counter to 0 but render N dots, the win check is off by N. Mitigation: explicit init `this.count = prefillCount`.
- **Tap zone collisions:** prefilled dots shouldn't be re-tappable in a way that confuses the kid. Mitigation: confirm tap zones add to the count whether or not a prefilled dot is in that cell (each cell holds one dot anyway, so this should be naturally handled).

#### Test plan

- TypeScript build clean.
- Manual: round with `prefillCount = 4`, target = 7. Three taps fill three more cells, win fires.
- Manual: round with `prefillCount = 0` — behaves like today.
- Manual: round with `prefillCount = 9`, target = 10 — one tap and win.

#### Estimated effort

**~1 hour** (clean plain HTML/JS, no Phaser scene state to wrangle).

#### Unlocks

- K.CC.A.2 PRIMARY engine `number-frames` flips from "conditional — needs counting-on mode" to clean.

### Fix 5 — `free-collect` pre-stocked-field mode

**Engine in plain English:** Free-collect is the Phaser engine where kids tap empty space and a dot/object appears where they tapped — like dropping marbles into a tray. Today every round starts with an empty field. We're adding pre-stocked: the field starts with N objects already present, and the kid's taps count as N+1, N+2, etc. Same idea as Fix 4 but for the free-form field instead of the structured ten-frame.

#### Scope

- **Add** a `prefillCount: number` config to `FreeCollectScene`. Default 0.
- **At scene start:** seed `currentTotal = prefillCount`. Render `prefillCount` dots into `_collectedObjs` at randomized field positions before any taps are taken.
- **Each tap:** increments `currentTotal` by 1 (existing behavior).
- **Stays the same:** tap-to-add behavior, win condition based on `currentTotal === target`, visual style of dots.
- **New game option ID** so Builders can select pre-stocked mode.

#### Files to touch

- `src/lib/game-engines/collect-manage-phaser.ts` — `FreeCollectScene` accepts `prefillCount` in config. Seed state and pre-render dots in `create()` (or wherever scene init happens) before tap handlers go live.
- `src/lib/game-engines/game-option-registry.ts` — new option `free-collect-count-on` (or similar) with display name and description.

#### Implementation steps

1. Read `FreeCollectScene` to map current state and dot-spawn flow.
2. Add `prefillCount?: number` to the scene's config. Default 0.
3. In scene `create`, before binding tap handlers: for `i = 0` to `prefillCount - 1`, spawn a dot at a randomized position (use the same spawn helper taps use). Push each into `_collectedObjs`. Set `this.currentTotal = prefillCount`.
4. Confirm tap handler still works — each tap adds one to `currentTotal`, win fires at target.
5. Register new option ID in the registry; wire option-ID → scene config.

#### Risks

- **Prefilled dot collisions:** randomized positions could overlap. Mitigation: use the existing collision-aware spawn helper if there is one; otherwise minimum-distance check.
- **Init order:** if dots get rendered AFTER tap handlers bind, a fast kid could double-tap and break the count. Mitigation: prefill first, bind taps last.
- **Visual jarring:** field appearing fully-stocked could surprise a kid expecting an empty field. Mitigation: that's the intended behavior — Builder copy should set the expectation ("9 marbles already here; tap to make it 10").

#### Test plan

- TypeScript build clean.
- Manual: scene with `prefillCount = 5`, target = 8. Three taps, win fires.
- Manual: scene with `prefillCount = 0` — identical to today.
- Manual: scene with `prefillCount = 9`, target = 10 — one tap and win.
- Manual: visual check — prefilled dots match tapped dots in style.

#### Estimated effort

**1–2 hours** (Phaser scene state seeding, registry, manual testing).

#### Unlocks

- K.CC.A.2 PRIMARY engine `free-collect` flips from "conditional — needs pre-stocked field mode" to clean (third of the three engines that need fixing for K.CC.A.2).
